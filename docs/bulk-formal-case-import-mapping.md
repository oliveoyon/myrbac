## Recommended Import Approach

Use `data.csv` as the working bulk import file. It has system-style headers, no duplicate headers, and already includes `district_id` and `pngo_id`.

The earlier `dataupload.csv` can remain as a human/reference file, but it should not be the main import file because it has repeated labels such as `Date (M-D-Y)`.

Recommended flow:

1. User uploads `data.csv` or the same structure as XLSX.
2. System reads by exact system header name.
3. System validates district/PNGO scope from `district_id` and `pngo_id`.
4. System defaults or normalizes `status = 2` if the business decision is to count imported historical data immediately.
5. System preserves phones and identity numbers as text.
6. System validates all rows before insert.
7. Import should be all-or-nothing per batch.

For 13,000+ rows, import by district/PNGO batch first. Test with 20 rows, then 500 rows, then larger batches.

## Implementation Notes For Later

- The current import template expects system field names, but the user file uses human labels. A new mapper should convert this friendly CSV into system field keys before validation.
- Because many headers repeat, `WithHeadingRow` alone is not safe for this CSV. Use raw rows and map by index.
- Dates should support values like `1-Jul-24`, Excel serial numbers, and `M-D-Y` style values.
- Phone and NID fields must be treated as strings to avoid losing leading zeros or converting large NID numbers into scientific notation.
- All rows should be validated before any insert.
- For production import, log the uploader, filename, row count, inserted count, failed count, and validation messages.
